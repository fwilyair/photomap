# 设计方案：修复轨迹线数值爆炸问题 (2026-03-20)

## 1. 问题描述
轨迹生成的 `Centripetal Catmull-Rom` 算法在处理轨迹的起止点时，因为采用了原地复制 $P_0$ 和 $P_n$ 的方法（即 `[P0, P0, P1...]`），导致在极坐标下产生分母接近于零的计算，进而导致坐标值被放大数万倍，产生横穿整个地图的橙色“长直线”。

## 2. 解决方案：虚拟切线外推 (Vector Extrapolation)
我们将不再使用重复的点，而是基于轨迹的方向计算两个“虚拟引点”：

### 2.1 起始段优化
*   计算从 $P_1$ 到 $P_0$ 的向量。
*   在 $P_0$ 外部外推约 25% 长度的位置创建一个点 $P_{head}$。
*   $P_{head} = P_0 - (P_1 - P_0) \times 0.25$。
*   用 $P_{head}$ 替换原本数组中第一个重复的 $P_0$。

### 2.2 结束段优化
*   同理，计算从 $P_{n-1}$ 到 $P_n$ 的向量。
*   在 $P_n$ 外部外推约 25% 长度的位置创建一个点 $P_{tail}$。
*   $P_{tail} = P_n + (P_n - P_{n-1}) \times 0.25$。
*   用 $P_{tail}$ 替换原本数组中最后一个重复的 $P_n$。

### 2.3 数值稳定性增强
*   在 `getT` 函数中，将 `max(a, 0.0001)` 调整为 `max(a, 1.0)`。由于在 `MKMapPoint` (Mercator 投影点) 空间，1.0 只是很小的几个像素宽度，这能更强力地防止任何距离过近导致的除零错误，而不影响视觉平滑度。

## 3. 测试验证 (Verification Plan)
1.  **双点边界测试**：构造只有 2 个脚印点（例如相距 2km）的数据，验证起止点是否正常。
2.  **极近距离测试**：构造 2 个极其接近（例如 50m）的数据点。
3.  **视觉回归测试**：确认在 3D 视角（Follow 镜头）下，转弯处是否依然保持平滑。

## 4. 影响范围
*   `PhotoMapViews.swift`：修改 `SplineRouteBuilder` 结构体代码。
*   无其他 UI 组件影响。
